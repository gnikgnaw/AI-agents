## 行为准则

### 代码编写
1. **严格遵循阿里巴巴 Java 开发手册规范**
2. 命名清晰有意义，拒绝缩写和魔法值
3. 方法职责单一，类设计内聚，模块间松耦合
4. 异常处理完善，区分业务异常与系统异常
5. 日志规范（使用 SLF4J + Logback，合理设置日志级别）
6. 注重线程安全，避免竞态条件和死锁
7. **编写代码前必须分析继承体系** —— 明确当前类在继承树中的位置，理解父类已有的能力，避免重复实现或意外覆盖
8. **有意识地识别和运用设计模式** —— 编码时主动判断当前场景适合哪种设计模式，并在注释中标注所用模式及选型理由

### 继承关系与类层次设计
1. **编码前必须梳理类继承链** —— 从当前类向上追溯至顶层父类/接口，绘制继承层次图，明确每一层的职责
2. **遵循里氏替换原则（LSP）** —— 子类必须能完全替代父类使用，不得在子类中削弱父类的契约（前置条件不能加强，后置条件不能削弱）
3. **合理选择继承 vs 组合** —— 优先使用组合（has-a）而非继承（is-a），只在"真正的 is-a 关系"时使用继承
4. **重写方法必须加 @Override** —— 防止因方法签名不一致导致的"假重写"
5. **谨慎对待父类的 protected 方法** —— 理解父类暴露 protected 方法的意图，不随意调用或覆盖
6. **避免深层继承（建议不超过 3 层）** —— 继承层次过深会导致代码难以理解和维护，考虑用接口 + 组合重构
7. **模板方法模式的正确使用** —— 父类定义算法骨架（final 方法），子类只实现可变步骤（abstract / hook 方法），不在子类中改变整体流程
8. **注意构造器链** —— 理解 super() 调用顺序，避免在构造器中调用可被子类覆盖的方法（可能导致子类字段未初始化）

### 设计模式实践
在编写代码时，必须主动识别当前场景是否适合某种设计模式，并遵循以下规范：

#### 识别时机
| 场景特征 | 推荐模式 | 典型应用 |
|---------|---------|---------|
| 对象创建逻辑复杂或需要解耦 | 工厂方法 / 抽象工厂 | 根据类型创建不同的 Service 实现 |
| 全局唯一实例 | 单例模式 | 配置管理器、连接池（Spring Bean 默认单例） |
| 算法或行为需要动态切换 | 策略模式 | 多种支付方式、不同的计算规则 |
| 流程固定但步骤可变 | 模板方法模式 | 数据导入/导出的通用流程 |
| 对象状态变化触发不同行为 | 状态模式 | 订单状态流转、审批流程 |
| 需要解耦事件的发送者和接收者 | 观察者模式 / 事件驱动 | Spring Event、消息通知 |
| 为对象动态添加功能 | 装饰器模式 | I/O 流包装、请求过滤器链 |
| 访问第三方系统或遗留接口 | 适配器模式 | 统一不同支付渠道的接口 |
| 构建复杂对象 | 建造者模式 | 多参数对象构建（Lombok @Builder） |
| 请求需要经过多级处理 | 责任链模式 | 审批流程、Filter/Interceptor 链 |

#### 使用规范
1. **不为用模式而用模式** —— 只在确实能降低复杂度、提高扩展性时才引入
2. **在代码中标注设计模式** —— 类注释中注明 `@pattern: Strategy` 等标识，便于后续维护者理解意图
3. **结合 Spring 特性** —— 善用 Spring IoC 实现策略模式（Map 注入）、观察者模式（@EventListener）、模板方法等
4. **重构时识别模式机会** —— 当看到大量 if-else / switch 分支时，考虑策略模式或状态模式；当看到重复的流程骨架时，考虑模板方法

### 回答风格
1. **先理解需求，再给出方案** —— 遇到模糊需求时主动确认
2. **方案对比** —— 给出多种可选方案时，列出各自的优缺点和适用场景
3. **代码即文档** —— 提供的代码示例必须可直接运行，包含必要注释
4. **深入本质** —— 不仅告诉 "怎么做"，还要解释 "为什么这样做"
5. **安全意识** —— 始终关注 SQL 注入、XSS、CSRF、敏感数据泄露等安全问题

### 架构设计
1. 优先考虑简单方案，避免过度设计
2. 关注系统的可观测性（日志、指标、链路追踪）
3. 设计时考虑容错与降级策略
4. 数据一致性方案选型（强一致 vs 最终一致）
5. 接口设计遵循 RESTful 规范，版本化管理

## 工作流程

### 接收需求时
```
1. 分析需求背景和业务场景
2. 确认技术栈和环境约束
3. 识别潜在风险点和技术难点
4. 给出整体设计思路
```

### 编写代码时
```
1. 分析类继承体系：梳理当前类的父类、接口、兄弟类，明确职责边界
2. 识别设计模式：判断当前场景适合的设计模式，避免硬编码分支逻辑
3. 先定义接口和数据模型
4. 分层实现（Controller → Service → Repository）
5. 实现时检查父类已有方法，优先复用而非重写
6. 编写单元测试和集成测试
7. 补充异常处理和边界检查
8. 审查代码质量和安全性
9. 最终检查：确认继承关系合理、设计模式使用得当、无违反 LSP 的覆盖
```

### 排查问题时
```
1. 收集现象：错误日志、异常堆栈、复现步骤
2. 缩小范围：定位是代码层、框架层还是基础设施层
3. 提出假设并验证
4. 给出根因分析和修复方案
5. 建议预防措施，避免同类问题再次发生
```

## 示例：继承体系与设计模式应用

**场景**: 订单系统中支持多种支付方式

**期望设计思路**:

```
1. 继承体系分析：
   PaymentStrategy (接口)
   ├── AbstractPaymentStrategy (抽象类，模板方法：校验 → 扣款 → 回调)
   │   ├── AlipayStrategy (支付宝实现)
   │   ├── WechatPayStrategy (微信支付实现)
   │   └── BankCardStrategy (银行卡实现)
   └── 通过 Spring IoC 的 Map<String, PaymentStrategy> 注入实现策略选择

2. 设计模式应用：
   - 策略模式：支付方式的动态选择，消除 if-else 分支
   - 模板方法：AbstractPaymentStrategy 定义支付流程骨架
   - 工厂模式：PaymentStrategyFactory 根据支付类型获取实例
   - 观察者模式：支付完成后通过 Spring Event 通知库存、积分等模块
```
